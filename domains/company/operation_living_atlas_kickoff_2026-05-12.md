# Operation Living Atlas Kickoff

## Purpose

Kick off the next Operation Living Atlas work unit by confirming the workflow, agent lanes, first model packet, and queue-table posture before any structured records, public pages, or app workflows change.

This kickoff is a Company planning artifact. It does not create Planner/Supabase writes, website behavior changes, app behavior changes, public commitments, launch targets, sponsor obligations, financial commitments, or roadmap commitments.

## Confirmed Operating Order

Operation Living Atlas is the evidence and record-building program. Operation Living Web is the public relationship experience that should consume curated Atlas outputs.

For the first model packet, use this order:

1. Select one bounded source set.
2. Build the source packet before writes.
3. Extract facts and keep evidence separate from interpretation.
4. Route each fact by publicness tier.
5. Let Research validate species, ecological interpretation, uncertainty, and claim safety.
6. Prepare proposed structured writes only after the packet is coherent.
7. Seek explicit approval before any Supabase, Planner, App, Web, or public-surface write.
8. Verify approved writes by reading records back.
9. Review Web/public rendering only after curated records exist.
10. Use the completed packet as the model for later Atlas and Living Web work.

## Queue Table Decision

Do not create a dedicated Living Atlas queue table for the first packet.

Use Markdown source packets plus existing Planner tasks for the pilot. The first packet should prove the real workflow before App or schema work is considered.

Revisit a dedicated queue table only if at least one of these becomes true:

- Multiple packets are active at once and status becomes hard to trust.
- Handoffs between Content, Research, Web, App, and Company are getting lost.
- Packet status needs filtering, assignment, approvals, or repeatable dashboards.
- Proposed writes need a structured review queue before Supabase changes.
- App needs an operator workflow because the manual packet process is repeatedly useful but too slow.

If a table is needed later, it should be App-owned schema/runtime work with Company defining governance semantics. Likely fields would include source type/id, packet status, primary subject, affected subjects, owner/lead, confidence, publicness tier, proposed write summary, approval state, linked records, blockers, and closeout notes.

## Review Rhythm

For the pilot, review after each packet stage:

- Source packet drafted.
- Evidence extracted.
- Publicness tiers assigned.
- Proposed writes prepared.
- Approved writes verified.
- Public rendering reviewed.
- Packet closed.

After the first packet closes, Company should decide whether the next packet should be another Lake Post-Seal species dossier, another published video, a biome arc, or a media folder.

## Approval Gates

Explicit approval is required before:

- Supabase writes.
- Planner task/project changes.
- App workflow, schema, migration, queue, or editor work.
- Website/public route behavior changes.
- Public claim changes that imply stronger certainty than Research supports.
- Business, public, financial, legal, sponsor, launch, or roadmap commitments.

## Current Kickoff Outcome

Operation Living Atlas can proceed with the Lake Post-Seal model packet using the existing source-packet workflow. No dedicated queue table is needed yet. App workflow support should remain deferred until the first packet proves the process and exposes real operator friction.
